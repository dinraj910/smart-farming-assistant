Smart Farming Assistant — Flow Blueprint

App Flow Blueprint

How the app should work,
screen by screen.

This document sketches the full working flow of the AI-Powered Smart Farming Assistant — from the moment a farmer opens the app to the moment they act on a recommendation. It covers the navigation map, every core screen as a low-fidelity wireframe, the decision logic behind each feature, and which pieces to build first. Read it top to bottom like a build plan, not a mood board.

platform Mobile-first (React Native + Expo), Web as secondary
geography Kerala-focused, broadened per-feature
innovation Intercropping + Market Analysis
How to read this: every screen and flow step is tagged MVP / Stretch / Advanced / Future. Build MVP first, end to end, before touching anything else — a complete simple app beats an incomplete impressive one at demo time.

06

Build Priority Roadmap

What to build first, what to add next, and what to leave for the report's "future scope" section.

Priority What Build When Note
MVP Login, Home, Crop Input/Result, Disease Capture/Result, Weather, Alerts Implementation Plan — Phases 4, 5, 7, 9 This alone is a complete, demoable app. Do not start anything else before this works end to end.
Innovation Intercropping suggestion, Kerala plantation-crop disease recognition, Market price + cross-state comparison Right after MVP is stable This is what your professor asked you to focus on — build this second, not last, so it isn't rushed.
Stretch "Why this crop" explanation, disease severity meter, profit comparison, rainfall trend, Profile/My Farms If ahead of schedule Add only after MVP + Innovation are both working reliably in a live demo.
Future IoT sensor integration, multi-language voice assistant, satellite imagery, expert consultation booking Not this semester Mention explicitly in the report as future scope — good answer material for viva.
If you only remember one thing from this document: build the MVP loop fully working first (Home → one feature → result → back to Home), then layer the Innovation features on top of a working skeleton — not the other way around.

08

ML Models to Build

Three real models, real free datasets, and where to start from instead of training from zero. Intercropping isn't listed — it's a rule-based lookup table, not a trained model.

Model 1 · Build First

Crop Recommendation

Classical ML · multi-class classification (Random Forest / Gradient Boosting)

Dataset

Crop Recommendation Dataset (Kaggle)
2,200 rows · 22 crops · N-P-K, temp, humidity, pH, rainfall
kaggle.com/datasets/atharvaingle/crop-recommendation-dataset

Pretrained model

None needed — tabular data, trains from scratch in minutes.

⚠ Kerala crop-list gap
Dataset has rice, banana, coconut, coffee, cotton, jute — but no pepper, cardamom, rubber, tapioca, or arecanut. Either scope recommendations to the overlapping subset, or supplement with extra rows built from KAU package-of-practices ideal ranges.
Model 2 · Build Second

Yield Prediction

Classical ML · regression (Random Forest / XGBoost Regressor)

Dataset

Agricultural Crop Yield in Indian States (Kaggle)
1997–2020 · filter to Kerala rows
kaggle.com/datasets/akshatgupta7/crop-yield-in-indian-states-dataset

Pretrained model

None needed — trains from scratch, reuses Model 1's data pipeline.

⚠ Small filtered sample
Filtering to Kerala-only rows shrinks the dataset a lot. Expect wider error margins — report a confidence range, not a single precise number.
Model 3 · Longest Lead Time

Disease Detection (CNN)

Deep Learning · image classification, transfer learning

Datasets

PlantVillage (Kaggle) · 38 classes, common crops
kaggle.com/datasets/emmarex/plantdisease
New Plant Diseases Dataset (Augmented)
kaggle.com/datasets/vipoooool/new-plant-diseases-dataset
Coconut Leaf Dataset for Pest ID (Kerala-relevant)
kaggle.com/datasets/shravanatirtha/coconut-leaf-dataset-for-pest-identification

Pretrained model

Base: google/mobilenet_v2_1.0_224 (ImageNet)
huggingface.co/google/mobilenet_v2_1.0_224
Shortcut: already PlantVillage-tuned checkpoint
huggingface.co/linkanjarad/…plant-disease-identification
⚠ No public pepper/rubber dataset
Pepper and rubber leaf-disease images don't exist as a clean public dataset. This is the self-collection work flagged since the Requirement Collection doc — budget real field-visit time for it.

Order of building the models

1st Crop Recommendation
~3–5 days · validates pipeline
2nd Yield Prediction
~3–5 days · reuses pipeline
3rd Disease Detection CNN
~3–4 weeks training + tuning
The order above is for actual model training. The Kerala leaf-photo collection for Model 3 needs to start in Week 1 regardless — it's the slowest-moving resource in the whole project, not the model code itself.

09

Release Phases — each one a working product

Four phases. Each ends with something a farmer could actually use — not partial, internal-only progress. If time runs out, whichever phase you finished last is still a demoable product.

01 Weeks 1–6

Foundation Release

Login · Home · Soil Intelligence · Crop Recommendation (live model)

🧑‍🌾 A farmer can log in, enter soil values, and get a real ML-backed crop suggestion.
02 Weeks 7–10

Innovation Release

+ Yield Prediction · + Intercropping Suggestion · + Weather Intelligence

🧑‍🌾 A farmer gets a full recommendation with expected yield and an intercropping plan, plus a weather check before planting.
03 Weeks 10–13

Diagnosis + Market Release

+ Disease Detection (Kerala crops) · + Market Intelligence (Kerala + cross-state) · + Unified Alerts

🧑‍🌾 A farmer can diagnose a sick crop from a photo, check today's price before selling, and get notified automatically. This is your true finish line — full MVP + both innovation features, submission-ready.
04 Weeks 13–16

Polish Release

+ "Why this crop" explanation · severity meter · profit comparison · Profile · final QA + deployment

🧑‍🌾 Same product, refined — this is your final demo build, not a requirement to reach.
If the timeline slips anywhere, protect Phase 3 above everything else. Phase 4 is genuinely optional — a judge sees a complete, working, innovative app either way.
